--- Page stream 9 ---
 
### Authentication Service 
- Email + password login with JWT (access: 1h, refresh: 12h, session expires 12h) 
- Google OAuth2 (via `social-django`) - stores name + email from Google 
- Registration: unique username, valid email, password, confirm password -> OTP via email -> JWT 
- Forgot password: email -> OTP -> reset -> JWT 
- OTP: 6-digit, 10-minute expiry, sent via Gmail SMTP (free) using Django's email backend 
- Throttling: 100 login attempts/hour per IP -> 1-hour block (Redis-backed) 
- Change password: requires OTP re-verification 
 
### Social Service 
- Add friends via search (fuzzy using `django-fuzzywuzzy` or `pg_trgm` PostgreSQL extension - free) 
- Friend suggestion based on Loyalty Score: `(total_given - total_lent) / total_transactions` 
- Cursor pagination: 10 friends per page, "Show More" fetches next 10 
- Friend request system: send -> pending -> accept/reject 
- Loyalty score visible only to the user themselves, not friends 
 
### Transaction Service 
- Ledger-based: every give/lend is a transaction entry 
- Net balance per friendship: auto-calculated as `SUM(given) - SUM(received)` 
- Negative net = you owe; Positive net = friend owes 
- Optional due date per transaction 
- Transactions auto-deleted(soft delete) after 30 days from user history but exist in DB for later audit(Celery beat) 
- Confirmation flow: lender logs transaction -> borrower notified -> borrower confirms/denies 
- Idempotency key on transaction creation to prevent duplicates 
 
### Notification Service 
- WebSocket via Django Channels + Redis channel layer 
- 3 notification types: 
 1. Pending lend confirmation (with Agree/Disagree buttons) 
 2. Payment received acknowledgment 
 3. Due date reminder (repeats on every login until cleared) 
- Offline: service worker caches notifications in IndexedDB; sync on reconnect 
 
### Admin Panel 
- Admin can: create, read, delete users 
- All financial data stored AES-256 encrypted in DB (using `django-cryptography`) 
- Admin sees only: username, loyalty score, friend list 
 
### PWA / Offline Support 
- Service Worker caches: friend list, notifications, home page charts

- IndexedDB for local notification queue

 3. Detailed System Design 
 
### Architecture Pattern 
**Event-Driven + Social Graph + Mission-Critical** 
- Event-Driven: transaction events emit to Redis pub/sub -> notification service consumes 
- Social Graph: friend relationships modeled as a graph in PostgreSQL (adjacency list, no separate graph DB needed at this scale) 
- Mission-Critical: Circuit breaker pattern via `pybreaker` library on all inter-service calls; Mutex via Redis `SET NX` for idempotent operations; Optimistic Locking via `select_for_update()` on balance updates 
 
### Services Breakdown 
 
| Service | Framework | Port | 
|---|---|---|---|

Inter-service communication: grpc + Redis pub/sub (events) + direct HTTP via internal Docker network (synchronous lookups). 
 
### Security Implementation 
 
| Threat | Mitigation | 
|---|---| 
| User enumeration | Generic error messages on auth failures | 
| BOLA/IDOR | Object-level permission check on every ViewSet action | 
| Brute force | Redis-backed rate limiter: 100/hr -> 1hr block | 
| Mass assignment | DRF serializers with explicit `fields` - no `__all__` | 
| SQL injection | Django ORM only, parameterized queries | 
| Excessive data exposure | Serializers return only necessary fields | 
| Sensitive errors | `DEBUG=False` in prod; custom error handlers | 
| Encryption | AES-256 for financial fields via `django-cryptography` | 
| Secrets | `.env` files + Render environment variables | 
| CORS | Whitelist only frontend domain | 

--- Page stream 13 ---
| Clickjacking | `X-Frame-Options: DENY` header | 
| SSRF | No user-supplied URLs fetched server-side | 
| Race conditions | `select_for_update()` + Redis `SET NX` mutex | 
| Session hijacking | Short-lived JWTs + refresh token rotation | 
| DoS/DDoS | Nginx rate limiting + Render's DDoS protection | 
| Docker security | Non-root user in Dockerfile, minimal base image (`python:3.12-slim`) | 
| Dependency scanning | `pip audit` in CI | 
| Log rotation | Configured in Django logging + Render log drain | 
| Security headers | `django-csp` + `django-axes` | 
 
### Performance & Scalability 
 
- Cursor pagination (keyset) on all list endpoints - no OFFSET 
- `select_related()` and `prefetch_related()` to minimize DB hits 
- Redis caching: friend list (TTL 5min), loyalty scores (TTL 15min) 
- `__slots__` on Python dataclasses for serialization helpers 
- List comprehensions over loops in all utility functions 
- Idempotency keys stored in Redis (TTL 24h) 
- Circuit breaker (`pybreaker`) on notification dispatch 
- Optimistic locking on balance update: check version field before write 
- Try to keep the codes time complexity near O(logn)

5. gRPC Integration 
gRPC is used for all synchronous inter-service calls where one service needs a real-time response from another. Redis pub/sub handles fire-and-forget events. 
When gRPC is used: 
Ǆ 
Gateway -> Auth service: token validation on every request 
Ǆ 
Gateway -> Social service: friend existence check before transaction creation 
Ǆ 
Transaction service -> Notification service: trigger notification on transaction confirm/deny 
Ǆ 
Social service -> Auth service: fetch user display info (username) for friend suggestions 
Proto file location: shared/proto/socchoproto - compiled for each service using grpcio tools. 
Sample proto definition: 

--- Page stream 19 ---
syntax = "proto3"; 
package soccho; 
 
service AuthService { 
 rpc ValidateToken(TokenRequest) returns (TokenResponse); 
} 
message TokenRequest { string token = 1; } 
message TokenResponse { string user_id = 1; bool valid = 2; } 
 
service NotificationService { 
 rpc SendNotification(NotificationRequest) returns (NotificationResponse); 
} 
message NotificationRequest { 
 string recipient_id = 1; 
 string type = 2; 
 string payload = 3; 
} 
message NotificationResponse { bool delivered = 1; } 
 
Each service runs a gRPC server on an internal Render private port (not publicly exposed). The FastAPI gateway is the only gRPC client facing the outside. Internal service-to-service gRPC uses Render's private network - no TLS overhead on internal calls.

10. Scalability, Speed, Consistency, Availability  Partition Tolerance 
CAP positioning: AP with tunable consistency 
Soccho prioritizes Availability and Partition Tolerance. For financial balance reads, consistency is enforced at the DB layer using select_for_update() - this is the one operation where the system blocks briefly to ensure correctness. Everything else (friend lists, notifications, search history) is eventually consistent via Redis cache. 
Speed 
Ǆ 
Cursor (keyset) pagination everywhere - no OFFSET, no full table scans 
Ǆ 
pg_trgm GIN index on username for fuzzy search without scanning every row 
Ǆ 
select_related() / prefetch_related() - zero N+1 queries 
Ǆ 
USE DB triggers for repeated queries 
Ǆ 
Redis caches: friend list TTL 5 min, loyalty scores TTL 15 min, search history 24 hr 
Ǆ 
FastAPI gateway is async - handles hundreds of concurrent connections with a single worker 
Ǆ 
Graphene batches DB queries using graphene django's DataLoader (avoids N+1 in GraphQL resolvers) 
Availability 
Ǆ 
Each service is independently deployed on Render - one service down does not cascade 
Ǆ 
Circuit breaker (pybreaker) on all gRPC calls: if Auth gRPC is unreachable, gateway falls back to cached JWT validation for 30 seconds 
Ǆ 
Celery workers on Render background worker instances - separate from web processes, so a queue backlog doesn't affect HTTP response times 
Partition tolerance 
Ǆ 
Redis pub/sub for events means services do not need synchronous coupling for notifications 
Ǆ 
Transaction service writes to its own DB first, then emits Redis event -> Notification service consumes asynchronously. If Notification is partitioned, the event queues in Redis and is retried when the service recovers 

--- Page stream 34 ---
 
Consistency 
Ǆ 
select_for_update() on balance row during transaction confirmation - prevents double-spend race condition 
Ǆ 
Idempotency key (Redis SETNX, 24h TTL) on transaction creation endpoint - duplicate network retries are safe 
Ǆ 
Optimistic locking (version field) on balances table as secondary guard

14. Soccho user guide 
Welcome to **Soccho**, your personal companion for managing money shared among friends. Whether you are splitting a lunch bill, lending a bit of cash to a colleague, or keeping track of who owes what after a group trip, Soccho simplifies the process with a visually engaging and secure interface. This guide will walk you through everything you need to know to master the app and keep your personal ledger organized. 
 
### **Getting Started** 
Your journey with Soccho begins with a simple and secure signup process. You can choose to create a unique account using your email address and a password or use your existing **Google account** for a faster experience. To ensure your account is protected from the very start, the system will send a **6-digit security code** (OTP) directly to your email inbox. Simply enter this code to verify your identity and unlock your dashboard. This step ensures that only you can access your financial records and personal history. 
 
### **The Dashboard and Visuals** 
Once you log in, you are greeted by a modern, **3D-inspired home screen** designed to give you an instant snapshot of your financial health. At the top, you will see animated **pie and bar charts** that visually summarize your current standing. These charts update in real-time as you log new transactions. Below the charts is your friends list, where each entry features a colorful "balance chip." We use a simple color-coding system: **green** indicates that a friend owes you money, while **red** indicates that you owe them. The appás "depth" and smooth transitions make navigating between your summary and individual friendship details feel intuitive and fluid. 
 

--- Page stream 46 ---
### **Finding Friends and the Loyalty Score** 
Connecting with people is the core of the Soccho experience. You can find friends using the search bar, which is smart enough to find names even if you make a small typo. For your convenience, the app **remembers your search history for at least 24 hours**, making it easy to return to a person you were recently looking for. 
 
When you search for friends, you might notice a **Loyalty Score**. This is a private rating-visible only to you-that helps you understand your financial relationship with others. It is calculated based on how reliably transactions are settled and how often you interact. Higher scores suggest a history of consistent and successful lending and borrowing, helping you decide how to manage your future interactions. 
 
### **Recording and Confirming Transactions** 
Logging a transaction is designed to be a collaborative process to prevent any "he-said, she-said" situations. When you give or lend money, you simply select the friend, enter the amount, and can even add an optional **due date** if you want a reminder later. 
 
Once you submit a transaction, it isn't set in stone immediately. It is sent to your friend as a **pending request**. They will receive a notification and can choose to "Agree" or "Disagree" with the entry. Once they confirm it, your mutual balance and the charts on your home screen update instantly. This ensures that both parties are always on the same page regarding the debt. 
 
### **Notifications and Reminders** 
Soccho keeps you informed through a dedicated notification drawer that you can slide open from the side of your screen. This drawer manages three main types of alerts: 
1. **Lending Confirmations:** Requests from friends to verify a new transaction. 
2. **Acknowledgements:** Confirmations that a payment you made has been received. 
3. **Due Date Reminders:** If a transaction has a due date, the app will send you a reminder so you never forget a deadline. 
 
### **Automatic Cleanup and Privacy** 
To keep your records relevant and clutter-free, Soccho automatically **removes transaction entries that are older than 30 days**. This encourages you to settle debts promptly and ensures your dashboard only shows your most current financial activities. Furthermore, your privacy is a top priority; your sensitive financial data is scrambled behind the scenes so that it remains private and secure from any unauthorized eyes. 
 
### **Using Soccho Anywhere** 
Life happens on the go, and you might not always have a perfect internet connection. Soccho is built to work **offline**. You can still browse your friend list and log transactions while youáre out and about. The app will securely store your actions and automatically **sync them with the server** the moment you are back online. Whether you are using the app on a mobile phone or a desktop, the experience remains fast, responsive, and reliable.

15. UI/UX Design 
 
**Philosophy:** Facebook-inspired layout, mobile-first, 3D card depth via `box-shadow` + `transform`, smooth transitions, no paid component libraries. 
 
**Color palette:** Deep indigo primary (`#4F46E5`), white surfaces, soft gray backgrounds, accent green for "owed to you", accent red for "you owe." 
 
**Key screens:** 
 
*Login/Register* - centered card on gradient background, animated field focus rings, Google OAuth button with icon, smooth slide-in transitions between login -> register -> OTP. 
 
*Home page (mobile)* - top: notification bell with badge count -> tapping slides in notification drawer from left covering 70% of screen. Below: summary pie chart (Chart.js, animated on load), bar chart (monthly), then friends list with avatar circles + net balance chip (green/red). Sticky bottom nav: Home / Friends / Profile. 
 
*Friendship page* - top card shows friend avatar + net balance in large text. Below: "Give money" form (amount + optional due date) with animated submit button. Transaction history as timeline cards that fade in on scroll. 
 
*Find Friends* - search bar with live results (HTMX `hx-trigger="keyup changed delay:300ms"`). Below: "Suggested Friends" sorted by loyalty score. Pagination via "Show More" button (HTMX `hx-swap="beforeend"`). 
 
*Notifications drawer* - slides in from left. Three visual types: orange pending (with Agree/Disagree chips), gray info, red due-date reminder. Due reminders persist with a pulsing red dot until cleared. 
 
*404 page* - animated SVG of a broken taka coin with bounce animation, "Lost your taka?" text, back-to-home button. 
 
**Animations:** CSS `@keyframes` only - no heavy libraries. Slide-in for drawers (200ms ease-out), scale-in for cards (150ms), number count-up on balance figures (JS `requestAnimationFrame`), skeleton loaders on data fetch. 
 
**3D feel:** Cards use `box-shadow: 0 4px 20px rgba(0,0,0,0.08)` + `transform: translateY(-2px)` on hover. Balance chip uses layered shadows for depth. Avatar circles use `ring` border with inset shadow.
